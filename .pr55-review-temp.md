## PR Review: feat(pi): add ensemble-pi package

Overall this is a well-structured PR that mirrors the opencode translation pattern cleanly. The 234-test suite is comprehensive and the generator architecture is solid. Below are the issues I found, grouped by severity.

---

### Must Fix

**1. `.beads/` runtime files committed to version control**

`.beads/dolt-server.pid`, `.beads/dolt-monitor.pid.lock`, `.beads/dolt-server.port`, `.beads/.local_version`, and the `.beads/dolt/` binary directory should not be in version control. PID files are stale on every clone; the Dolt binary database may be large/sensitive. Add these to `.gitignore` before merge.

Introducing Beads issue-tracking as an undeclared repo dependency (requires `bd` CLI and Dolt) with no mention in CLAUDE.md or README is also a contributor experience hazard.

**2. `packages/pi/plugin.json` references a non-existent `commands/` directory**

`"commands": "./commands"` is set in `.claude-plugin/plugin.json`, but `packages/pi/` has no `commands/` subdirectory. Claude Code plugin loading will silently fail or warn for every user who installs this plugin.

**3. `glob` used as an undeclared direct dependency**

`generator.ts` imports from `glob`, but `packages/pi/package.json` only has it transitively via `gray-matter`/`js-yaml`. If those transitive deps change their bundled glob version the API can break silently. Add `glob` to `dependencies` directly.

---

### Should Fix

**4. Package name mismatch: `@fortium/ensemble-pi` vs `ensemble-pi`**

`package.json` uses the scoped name `@fortium/ensemble-pi` but `marketplace.json` registers it as `ensemble-pi` (unscoped). All other marketplace entries match their npm package names exactly. This breaks `npm publish`/install symmetry.

**5. Version inconsistency: `1.0.0` vs monorepo `5.x.x`**

Both `plugin.json` and the marketplace entry use `"version": "1.0.0"` while every other ensemble package is `5.x.x&. Should be `5.1.0` to match the current monorepo version unless there is a documented reason for the divergence.

**6. `pi-generate.yml` workflow has no test step**

The dedicated Pi CI workflow only runs generation and drift detection, never `npm test --workspace=packages/pi`. Verify the existing `test.yml` covers this workspace; if not, add a test step.

**7. Dead code: `findFilesRecursive` in `generator.ts`**

This helper is defined but never called -- `discoverYamlFiles` always uses `glob`. Remove it.

---

### Consider Fixing

**8. `conditional`/`trigger` fields silently dropped in Pi output**

The new `--interactive` parameter in `fix-issue.yaml` uses `conditional: true` and `trigger:` fields on steps. These are not in the `CommandYaml` interface in `types.ts`, so the command transformer silently drops them. If intentional (Pi does not support conditional steps natively), add a comment in `command-transformer.ts` explaining why.

**9. `.claude/settings.local.json` has stale/broken entries**

The `allow` list contains broken shell fragments from interactive sessions. Since this file is committed it becomes repo-wide noise. Prune before merge.

**10. `--legacy-peer-deps` in CI is too broad**

`npm ci --legacy-peer-deps` in `pi-generate.yml` suppresses all peer dep warnings globally because `@mariozechner/pi-coding-agent` is not in the registry. A targeted `.npmrc` override would be more precise and avoid hiding future dependency conflicts.

**11. `generateAgentsMd` / skill self-writing is inconsistent with the rest of the generator**

The main `generate()` orchestrator skips `agents-md` in its write loop because `generateAgentsMd()` self-writes internally. Skills do the same. Commands and agent results are handled differently (orchestrator owns writes). Unifying to a single write path would make the flow easier to reason about and test.

**12. `marketplace.json` author schema inconsistency**

The `ensemble-pi` entry has `"author` with `"name"` and `"email"` but other entries only have `"name"` and `"url"`. Verify the marketplace schema does not require the `url` field.

---

### What's Well Done

- Translation-layer (not fork) is the right design -- YAML stays as the source of truth, mirroring what was done for OpenCode.
- 234 tests across 11 suites cover all transformer paths, CLI flags, and structural equivalence well.
- `AskUserQuestion` to `ask_user` global rewrite is clean; the extension registration in `ask-user.ts` is well-isolated with good interrupt/abort handling.
- Drift-detection CI workflow is a solid safeguard against generated artifacts going stale.
- `AGENTS.md` extraction from `CLAUDE.md` is a nice touch for Pi's agent mesh reference.
